## システムの概要

このシステムは、以下の3つの要素で構成されています：

1. **ルール (Rules)**: あなたの執筆スタイル、価値観、思考の癖を言語化した「憲法」
2. **知識 (Knowledge)**: あなたのインプットや過去のアウトプットである「記憶」
3. **指示 (Instructions)**: 具体的なタスクを依頼するための「命令」

### プロジェクト構造

```
ライティングシステム/
├── .cursor/                     # Cursor設定ディレクトリ
│   └── rules/                   # ルール定義
│       ├── 10_writing-style/    # ① 文体・スタイル設定
│       │   └── RULE.md
│       ├── 20_platform-rules/   # ② プラットフォーム特化ルール
│       │   └── note/
│       │       └── RULE.md
│       └── 30_content-quality/  # ③ コンテンツ品質基準
│           └── RULE.md
├── .cursorignore                # コンテキストから除外するファイル
├── README.md                    # 指示ハブ (プロンプトの起点)
├── MANUAL.md                    # このマニュアル
├── knowledge/                   # ナレッジベース (AIの記憶)
│   ├── style-samples/           # 過去の良質な記事 (3〜5本)
│   │   ├── article_1.md
│   │   └── ...
│   ├── inputs/                  # インプット情報 (書籍、イベントメモ)
│   │   └── (日付)_(テーマ名).md
│   └── outputs/                 # 生成された記事の保存場所
│       └── yyyy-mm-dd_タイトル/ # 記事ごとのフォルダ
│           ├── draft.md         # AIが生成した下書き
│           └── article.md      # 完成した記事
```

## 初期セットアップ

### ステップ1: プロジェクトの確認

プロジェクトが正しくセットアップされているか確認してください。上記のプロジェクト構造に従って、すべてのディレクトリとファイルが存在することを確認します。

### ステップ2: 文体サンプルの配置

`knowledge/style-samples/` ディレクトリに、あなたの文体が最もよく現れている過去の記事を3〜5本配置します。

**手順：**

1. あなたが過去に書いた記事の中で、「これは自分らしい」と胸を張れる記事を選びます
2. その記事の内容をMarkdown形式で `article_1.md` として保存します
3. 同様に、`article_2.md`, `article_3.md` なども作成します（3〜5本が推奨）

**重要：**
- ファイル名は `article_1.md`, `article_2.md` のように連番で命名してください
- 記事はMarkdown形式で保存してください
- できるだけ多様なテーマの記事を選ぶと、AIの学習効果が高まります

### ステップ3: ルールのカスタマイズ

`.cursor/rules/` 内の3つの `RULE.md` ファイルを、あなた自身の言葉で書き換えます。

#### 3-1. 文体ルールのカスタマイズ

`.cursor/rules/10_writing-style/RULE.md` を開き、以下の項目をあなた自身の言葉で記述します：

- **基本的な考え方**: あなたが文章を書く上で大切にしている価値観
- **具体的な文体ガイド**: 文末表現、接続詞の使い方、一人称の使い方など
- **避けるべき表現**: あなたが絶対に使いたくない表現や、AIが使いがちな無機質な表現

**カスタマイズのコツ：**
- 時間をかけて丁寧に記述してください。これが最も重要なファイルです
- 具体的な例を挙げると、AIの理解が深まります
- 「避けるべき表現」のセクションは、実際にAIが生成した記事をレビューしながら、徐々に追加していくのが効果的です

#### 3-2. プラットフォームルールの確認

`.cursor/rules/20_platform-rules/note/RULE.md` を開き、noteのMarkdown仕様に合わせて調整します。

現在の設定は基本的なものですが、必要に応じて以下を追加できます：
- 特定の見出し構造のルール
- 画像の配置ルール
- リンクの形式

#### 3-3. 品質基準の確認

`.cursor/rules/30_content-quality/RULE.md` を開き、あなたが求める品質基準を確認・調整します。

## 日常的な使い方

### ワークフロー概要

1. **執筆準備**: テーマを決め、参考情報を収集
2. **AIによるドラフト生成**: Cursorに指示を出して記事の下書きを作成
3. **レビューと仕上げ**: 生成された記事を人間がレビューし、修正・完成させる

### ステップ1: 執筆準備

#### 1-1. テーマ設定

書きたいテーマを決めます。

#### 1-2. 情報収集とインプットファイルの作成

そのテーマに関する参考情報（URL、メモ、関連データなど）を集め、`knowledge/inputs/` ディレクトリに新しいファイルを作成します。

**ファイル名の形式：**
```
(日付)_(テーマ名).md
```

**例：**
```
2025-01-15_AIライティング.md
2025-01-20_プロジェクト管理.md
```

**ファイルの内容：**
- 参考URL
- 読んだ本のメモ
- イベントの記録
- 関連データ
- その他の情報
- 伝えたいポイント

### ステップ2: AIによるドラフト生成

#### 2-1. README.mdを開く

プロジェクトのルートにある `README.md` を開きます。ここがAIとの対話の起点です。

#### 2-2. プロンプトの入力

`README.md` に記載されているテンプレートを参考に、AIにドラフト作成を指示します。

**プロンプトの例：**

```markdown
# 記事生成プロンプト

## 指示
すべての.cursor/rules/内のルールを厳格に遵守し、以下の条件でnote記事のドラフトを作成してください。

## 条件
- **テーマ**: AIライティングの実践的なコツ
- **読者ターゲット**: ブログやnoteで記事を書いている人
- **記事のゴール**: AIを活用して自分らしい文章を書くための具体的な方法を理解できる
- **参考情報**: @knowledge/inputs/2025-01-15_AIライティング.md
- **参考文体**: @knowledge/style-samples/ (このディレクトリ内の全記事)

## 出力
- 記事を執筆したら、`knowledge/outputs` 内に対応するタイトル「yyyy-mm-dd_タイトル」というフォルダを作成する
- 執筆した記事は「draft.md」という名前で作成したフォルダ内に保存する
- 例：`knowledge/outputs/2025-01-15_AIライティング/draft.md`
```

**重要なポイント：**
- `@` を使って必要なファイルを正確にコンテキストに含めることが重要です
- テーマ、読者ターゲット、記事のゴールを明確に指定します
- 出力先のパスも正確に指定します

#### 2-3. AIの生成を待つ

Cursorが記事を生成するのを待ちます。生成には数分かかる場合があります。

### ステップ3: レビューと仕上げ

AIが生成した `draft.md` は、あくまで「優れた下書き」です。ここからあなたの手で魂を吹き込みます。

#### 3-1. ファクトチェック

AIが生成した情報の正確性を検証します。特に、数値データや専門的な内容については注意が必要です。

**チェックポイント：**
- 数値データの正確性
- 専門用語の使い方
- 引用元の確認

#### 3-2. 文体レビュー

`writing-style` ルールが守られているか確認します。AIが苦手とする微妙なニュアンスや、あなたなら使わない表現を修正します。

**チェックポイント：**
- 文末表現が統一されているか
- 接続詞の使い方が自然か
- 一人称の使い方が適切か
- 「避けるべき表現」が使われていないか

#### 3-3. 独自性の追加

あなた自身の具体的な経験談、失敗談、個人的な洞察などを追記します。ここが、記事に「あなたらしさ」を与える最も重要なプロセスです。

**追加すべき内容：**
- 具体的な経験談
- 失敗談や学び
- 個人的な洞察や意見
- 読者への具体的なアドバイス

#### 3-4. 最終稿の完成

修正が完了したら、内容を `article.md` として保存します。

**ファイル構造：**
```
knowledge/outputs/2025-01-15_AIライティング/
├── draft.md    # AIが生成した下書き
└── article.md  # あなたが完成させた最終稿
```

**重要：**
- 執筆は常に同一フォルダ内でドラフト⇒本記事の手順で進める
- `draft.md` を修正するのではなく、完成版を `article.md` として新規保存する

#### 3-5. noteへの投稿

完成した `article.md` の内容をコピーし、noteの編集画面に貼り付けて投稿します。

## ルールのカスタマイズ

### フィードバックループの重要性

レビューの過程で「AIはこういう表現を使いがちだな」といった気づきがあれば、それを `.cursor/rules/10_writing-style/RULE.md` の「避けるべき表現」に追加していきましょう。

この地道なフィードバックループが、AIをあなただけの最高のパートナーに育て上げます。

### ルールの更新手順

1. AIが生成した記事をレビューする
2. 気になる表現や、改善したい点をメモする
3. `.cursor/rules/10_writing-style/RULE.md` を開く
4. 「避けるべき表現」セクションに追加する
5. 必要に応じて、他のセクションも更新する

### 効果的なルールの書き方

- **具体的に記述する**: 「無機質な表現を避ける」ではなく、「『〜することができます』という表現は使わない」と具体的に記述する
- **例を挙げる**: 良い例と悪い例を並べて記述すると、AIの理解が深まる
- **理由を説明する**: なぜそのルールが必要なのかを説明すると、AIが文脈を理解しやすくなる

## トラブルシューティング

### よくある問題と解決方法

#### Q1: AIが生成した記事が、自分の文体と違う

**原因：**
- 文体サンプルが不十分
- ルールが具体的でない
- 参考ファイルがコンテキストに含まれていない

**解決方法：**
1. `knowledge/style-samples/` に、より多様な記事を追加する
2. `.cursor/rules/10_writing-style/RULE.md` をより具体的に記述する
3. プロンプトで `@knowledge/style-samples/` を明示的に指定する

#### Q2: AIが「避けるべき表現」を使っている

**原因：**
- ルールが明確でない
- ルールがコンテキストに含まれていない

**解決方法：**
1. `.cursor/rules/10_writing-style/RULE.md` の「避けるべき表現」セクションをより具体的に記述する
2. プロンプトで「すべてのルールを厳格に遵守する」と明示する
3. 生成後にレビューし、問題のある表現をルールに追加する

#### Q3: 生成された記事の品質が低い

**原因：**
- インプット情報が不十分
- テーマやゴールが不明確

**解決方法：**
1. `knowledge/inputs/` のファイルにより詳細な情報を追加する
2. プロンプトでテーマ、読者ターゲット、記事のゴールをより明確に指定する
3. 必要に応じて、プロンプトを複数回に分けて実行する（例：まず構成を作成、次に本文を生成）

#### Q4: ファイルがコンテキストに含まれていない

**原因：**
- `@` 記号の使い方が間違っている
- ファイルパスが間違っている

**解決方法：**
1. プロンプトで `@` を使ってファイルを明示的に指定する
2. ファイルパスが正しいか確認する（相対パスで指定）
3. `.cursorignore` でファイルが除外されていないか確認する

## まとめ

このシステムを使うことで、AIを「自分の分身」として育て、自分らしい文章を安定して再現できるようになります。

重要なのは、AIを単なるツールとして使うのではなく、対話とフィードバックを通じて育てていくという視点です。

最初はうまくいかないこともあるかもしれませんが、継続的にルールを更新し、サンプルを追加していくことで、AIは徐々にあなたの分身となり、あなたはより本質的な「何を伝えるか」という創造的活動に集中できるようになるでしょう。

Happy Writing! ✍️
